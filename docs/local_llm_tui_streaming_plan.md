# ローカルLLM向けGo言語TUIストリーミング実装計画

## ゴール
Go言語を用いたTUI（Terminal User Interface）アプリケーションにおいて、ローカルLLMからのストリーミング出力をスムーズかつ効率的にレンダリングするための最新構成とベストプラクティスを定義し、実装計画を立案する。

## 調査結果と最新構成

### 1. 推奨される技術スタック
*   **TUIフレームワーク**: **Bubble Tea** (Charmbracelet)
    *   The Elm Architectureベースの関数型かつステートフルなアプローチにより、非同期なストリーム処理との相性が非常に良い。
*   **テキスト・マークダウンレンダリング**: **Glamour** (Charmbracelet)
    *   スタイルシート駆動のターミナル向けマークダウンレンダラー。Bubble Teaのエコシステムと完全に統合されている。
*   **UIコンポーネント**: **Bubbles** (`viewport` コンポーネント)
    *   大規模なテキストを表示し、スクロールを管理・最適化するために必須。
*   **ローカルLLMクライアント**: **Ollama Go SDK** または `go-ollama`
    *   ローカルでのLLM実行環境のデファクトスタンダードであるOllamaと型安全に通信できる。

### 2. VISION.md 準拠のための UX / UI ポリシー

asqlは「データ観察のための軽量CLI」であり、分析の「主役」ではありません。LLM連携機能もこの哲学に従う必要があります。

1.  **AIを主役にしない (On-Demand Activation)**:
    *   チャット画面や入力プロンプトを常設せず、ユーザーが明示的に呼び出した時（例: エラー発生時に `?` を押す、複雑なクエリ上で `A` を押す等）のみ、別ペインとして補助的に表示する。
    *   必要な情報が得られたらすぐに閉じられるようにする（`Esc`キー等）。
2.  **思考を止めない非同期処理 (No UI Freezing)**:
    *   LLMからのストリーミング受信中であっても、メインの操作（スクロール、タブ切り替え、別クエリの実行）がブロックされない完全な非同期UIを実現する。
3.  **ノイズの排除**:
    *   アニメーション（ローディング時の過度なスピナーなど）は最小限に抑え、「生成中...」といった控えめな表示にとどめる。
    *   マークダウンのスタイルも、asql全体のミニマルなカラーテーマに馴染むようGlamourの設定を上書きして調整する。
4.  **Bring Data Philosophyへの合致**:
    *   ローカルLLM（Ollama等）を前提とすることで、外部へのデータ送信（スキーマやログの流出）を防ぎ、自己完結したセキュアな「持ち寄る」体験を維持する。

### 3. ストリーミングレンダリング最適化のベストプラクティス

1.  **非同期メッセージング (`tea.Cmd` とチャネルの活用)**:
    *   LLMからのストリーム受信（Goroutine）とUIの更新（メインループ）を分離する。
    *   ストリームのトークンを受信するたびに Bubble Tea の `tea.Msg` を発行し、`Update` 関数で状態を更新する。
2.  **バッファリングとデバウンス処理 (Throttling / Debouncing)**:
    *   LLMからのレスポンスは細かなトークン単位で高速に届くため、**毎トークンごとにGlamourでマークダウンのフルレンダリングを行うと高負荷（CPUスパイク、画面のちらつき）の要因**となる。
    *   一定単位（数ミリ秒間のバッファリング、または改行単位）でトークンをまとめ、まとめてUIの更新（再レンダリング）を行うことで負荷を軽減する。
3.  **Viewportによるクリッピング**:
    *   `bubbles/viewport` コンポーネントを使用し、画面に表示されている行のみをターミナルに描画する。
    *   自動スクロール処理（新しいコンテンツが追加されたら最下部にスクロールする）と組み合わせ、長文になってもパフォーマンスを維持する。

## 提案する実装アーキテクチャ

*   **Model**:
    *   蓄積された完全なレスポンステキスト (`string` または `strings.Builder`)
    *   `viewport.Model` コンポーネント
    *   ストリーミング状態フラグ (生成中/完了)
    *   **LLMペインの表示/非表示フラグ (VISION準拠: オンデマンド表示)**
*   **Update**:
    *   キーイベント (`?` や `A` など): LLMペインを開き、入力フォームにフォーカス、または裏側のエラー内容で即座にプロンプトを送信する。
    *   `StreamTokenMsg(string)`: LLMからの新しいトークンを受け取り、内部テキストバッファに追加。レンダリングタイマーが発火していれば `viewport` のコンテンツを更新し、下へスクロールするコマンドを発行。メインUIとの並行操作を妨げない。
*   **View**:
    *   通常は非表示。アクティブ時のみ、メインビューの横 (または下部) にスプリット表示されるラッパーUI。
    *   フォーマット済みの出力を提供するマークダウンの実体は `viewport.View()` 経由で描画。カラーテーマに合わせた控えめな装飾。

## 実装ステップ

*   [ ] **Step 1: プロジェクトの初期化と環境構築**
    *   必要な依存関係のインストール (`bubbletea`, `bubbles`, `lipgloss`, `glamour`, `ollama` SDK).
*   [ ] **Step 2: TUI基盤の構築 (オンデマンドUX)**
    *   既存のasqlのUIに干渉しないように、特定のキー入力で表示/トグル分離される新しいLLMペイン（入力用の `textinput` と出力用の `viewport`）を作成する。
*   [ ] **Step 3: ローカルLLM連携 (モックまたは小規模モデル)**
    *   Ollama APIに接続し、入力に対するストリーミングレスポンスを受け取る Goroutine と `tea.Msg` のブリッジを実装。
*   [ ] **Step 4: レンダーパフォーマンンスの最適化 (★重要)**
    *   入力をバッファリングする仕組みを導入し、Glamourによるレンダリングを `[一定間隔]` または `[改行・特定の区切り]` タイミングで行うように調整。
*   [ ] **Step 5: 動作検証とチューニング**
    *   巨大なプロンプト出力を試し、メモリ使用量と遅延（CPU使用率）を確認。必要であればバッファサイズや描画インターバルを調整。
